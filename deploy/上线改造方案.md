# 上线前改造方案

> **状态**:已批准,待实施
> **制定日期**:2026-06-21
> **目标**:把项目从"开发期可联调"推进到"可生产上线"

## 总体决策(已与产品方确认)

| 决策项 | 选择 | 说明 |
|--------|------|------|
| 部署形态 | **单机 Docker**(三件套)+ 图片存宿主机 | 容器化应用,业务数据(MySQL/图片)留宿主机挂载,重建容器不丢数据 |
| 短信服务商 | **阿里云号码认证服务 - 短信认证(PNVS)** | 免资质免审核,个人实名即可;赠送签名 + 登录/注册模板(100001),无需签名/模板报备 |
| 域名规划 | **同域不同路径**(`/` H5、`/admin/` Admin、`/api/` 后端) | 现阶段无域名先用 IP,域名到位后切子路径 |
| Cookie 策略 | 保持 `SameSite=Lax` | 无域名阶段;上 HTTPS+域名后切 `SameSite=None; Secure` |
| 文件存储 | **本地磁盘**(`./uploads` → 宿主机 `/data/petshop/uploads`) | 单机无需 OSS |
| 会话存储 | **Sa-Token 内存**(不接 Redis) | 单机重启最多让用户重登一次,可接受 |

所有改动分 4 个独立模块,可分批提交、分批验证。

## 模块 A:短信认证服务接入阿里云(后端)

> **决策变更(2026-06-29)**:本项目短信仅用于 H5 手机号登录核验(无通知/营销短信需求),故弃用"短信服务 SMS + 自研验证码"路线,改用号码认证服务下的"短信认证"。后者将验证码的生成、有效期、防刷、核验全部托管给阿里云,且**免企业资质、免签名/模板报备**(个人实名账号 + 赠送签名/模板即可),后端无需 Caffeine 缓存与自研防刷逻辑。前端 UI 与接口保持不变。

### A1. 引入依赖
`backend/pom.xml` 增加号码认证服务 Java SDK:
- `com.aliyun:alibabacloud-dypnsapi20170525`(号码认证服务 PNVS SDK;注意是 `dypnsapi` 不是 `dysmsapi`,两者分属不同产品、不同 endpoint)

**不再需要** `spring-boot-starter-cache` / `caffeine`:验证码的生成、存储、有效期、防刷全部由阿里云内置,后端无需本地缓存。

### A2. 新增短信认证抽象层
新建 `backend/src/main/java/com/petshop/order/sms/`:
- `SmsVerifyService.java` — 接口,定义两个方法:`void send(String phone)`(发送,验证码由阿里云系统生成)与 `boolean verify(String phone, String code)`(核验)
- `AliyunPnvsSmsService.java` — 阿里云 PNVS 实现(`@ConditionalOnProperty(name="sms.provider", havingValue="aliyun")`),`send` 调 `SendSmsVerifyCode`,`verify` 调 `CheckSmsVerifyCode` 并依据返回 `VerifyResult` 判定
- `LogSmsService.java` — 兜底实现(仅打日志,`verify` 对固定码 `1234` 放行,便于本地联调),`provider` 非 `aliyun` 时生效

这样在阿里云 AK/签名未就绪前用 `LogSmsService` 联调,AK 一配即切真实核验,**业务代码无需改动**。

### A3. 改造 AppAuthServiceImpl
- 删除 `codeStore`(`ConcurrentHashMap`)与固定 `1234` 整段内存验证码逻辑
- 注入 `SmsVerifyService`:`sendSmsCode(phone)` 直接委托 `smsVerifyService.send(phone)`
- `login(phone, code)` 改为:`smsVerifyService.verify(phone, code)` 通过后再走原来的 `appUserMapper.selectByPhone` + `StpUtil.login`,不通过抛 `BusinessException("验证码错误")`
- **防刷交给阿里云**:`SendSmsVerifyCode` 的 `interval`(发送冷却,秒)与 `valid_time`(有效期,秒)参数即控制同手机号发送频率与验证码时效,后端无需再写冷却/计数逻辑

### A4. 配置外部化
`application.yml` 新增 `sms` 命名空间:
```yaml
sms:
  provider: ${SMS_PROVIDER:log}      # log(默认兜底) / aliyun
  aliyun:
    access-key-id: ${ALIYUN_SMS_AK:}
    access-key-secret: ${ALIYUN_SMS_SK:}
    sign-name: ${ALIYUN_SMS_SIGN:}                  # 号码认证控制台"赠送签名",例如"速通互联验证码"
    template-code: ${ALIYUN_SMS_TEMPLATE:100001}    # 登录/注册模板,赠送,免审核
```
默认 `provider=log` 保证没配 AK 也能启动(安全降级);生产注入 `SMS_PROVIDER=aliyun` + AK/SK + 赠送签名名即可。签名与模板沿用阿里云赠送资源,**无需提报审核**。

**前端无需改动**:`/api/app/auth/sms-code` 与 `/login` 接口路径、入参(手机号 + 4 位验证码)、60 秒倒计时、错误处理全部就绪,只是后端从"内存固定 1234"变成"调用阿里云核验"。

## 模块 B:CORS / 密钥 / profile 外部化(后端)

### B1. Spring Profile 机制
- 新建 `application-dev.yml`(当前 application.yml 内容,本地开发默认)
- 新建 `application-prod.yml`(生产配置,所有敏感值走 `${ENV}`,无默认值/默认值留空)
- `application.yml` 保留为公共配置,新增 `spring.profiles.active: ${SPRING_PROFILES_ACTIVE:dev}`
- 现有 `@Value("${...:default}")` 保留,prod profile 覆盖默认值即可

### B2. CORS 外部化
改造 `CorsConfig.java`:
- 从 `application.yml` 的 `app.cors.allowed-origins`(List)读取
- 删除硬编码的 localhost/192.168
- dev profile 配 localhost:3000/3001 + 192.168.*.*;prod profile 配生产域名/IP(列表可为空,通过环境变量 `CORS_ALLOWED_ORIGINS` 注入)

### B3. 密钥外部化
- `app.webhook.aes-key` 默认值移除(或改为开发弱密钥),生产强制 `WEBHOOK_AES_KEY` 环境变量
- `DataInitializer.java` 的 admin/admin123 改为从环境变量读(`ADMIN_INIT_PASSWORD`,默认仍 admin123 但打 WARN 日志提醒生产修改)

### B4. Cookie 配置
- prod profile 把 `sa-token.cookie.same-site` 改为 `None`(配合后续 HTTPS),现阶段 IP 访问保留 Lax
- 这一项**先不做强制改动**,等上域名 + HTTPS 时再切(避免现在改了反而 Cookie 丢)

## 模块 C:部署产物(从零搭建)

> **已迁移至 [`README.md`](./README.md)**
>
> 部署形态、Dockerfile、`docker-compose.prod.yml`、`nginx.conf`、`.env.prod` 清单与部署步骤,统一以 README 为权威。README 已基于实际代码校准,与原方案差异如下:
>
> - **修正 Admin 子路径 `base`**:原方案"vite base 不改"有误。Admin 挂 `/admin/` 时 `base` 必须为 `/admin/`,否则静态资源请求 `/assets/...` 会被 H5 的 `location /` 拦截,导致白屏。
> - **合并构建**:前端由"两个 Dockerfile"改为一个多阶段 Dockerfile,H5 与 Admin 输出到同一 nginx 容器。
> - **构建工具校准**:前端为 Vue + Vite 8 + pnpm,基础镜像用 `node:22-alpine`(Vite 8 要求 Node 20.19+),非 npm。
> - **nginx 配置**补全了 H5/Admin 图片共享、健康检查、gzip 等。
>
> 原 C1~C5 细节保留于 git 历史,不再在此维护。
