## 目标

把 H5 手机号登录从"内存固定 `1234` 验证码"改造成"阿里云号码认证服务(PNVS)短信认证"。验证码的生成、有效期、防刷、核验交给阿里云托管;开发期用 `LogSmsService` 兜底联调,生产 AK 到位即切真实核验。**前端零改动**。

## 设计要点

- **抽象层**:`SmsVerifyService` 接口 + 两个实现,按 `sms.provider` 条件装配,业务代码只依赖接口。
- **兜底优先**:`provider=log`(默认)时走 `LogSmsService`(打日志 + 固定码 `1234` 放行),保证 AK 未到位也能启动联调。
- **前端契约不变**:`POST /api/app/auth/sms-code`(入参 `phone`)、`POST /api/app/auth/login`(入参 `phone` + `code`)签名不变,`LoginPage.vue` 不动。
- **防刷不在后端写**:复用 `SendSmsVerifyCode` 的 `interval`(发送冷却)与 `valid_time`(有效期)参数。

## 实施步骤

1. **加依赖**:`pom.xml` 引入 `com.aliyun:alibabacloud-dypnsapi20170525`(确认是 `dypnsapi` 不是 `dysmsapi`)。
2. **建配置类**:`SmsProperties` 读取 `sms.provider` 与 `sms.aliyun.*`(AK、SK、sign-name、template-code)。
3. **建接口与实现**:
   - `SmsVerifyService`:`void send(String phone)` + `boolean verify(String phone, String code)`
   - `AliyunPnvsSmsService`:`send` 调 `SendSmsVerifyCode`,`verify` 调 `CheckSmsVerifyCode`,依据 `VerifyResult` 返回布尔
   - `LogSmsService`:日志打印 + `verify` 对 `1234` 放行
4. **配置外部化**:`application.yml` 增加 `sms` 块(见下方配置),默认 `provider=log`。
5. **改造 `AppAuthServiceImpl`**:
   - 删除 `codeStore` 与 `sendSmsCode` 内固定码
   - `sendSmsCode` → `smsVerifyService.send(phone)`
   - `login` → `smsVerifyService.verify(phone, code)` 通过后再查用户 + `StpUtil.login`,失败抛 `BusinessException("验证码错误")`
6. **本地联调**:`provider=log`,发短信接口成功、用 `1234` 登录通过。
7. **真实测试**(AK 到位后):`provider=aliyun`,真实手机收码、正误码分别验证。

## 关键配置

```yaml
sms:
  provider: ${SMS_PROVIDER:log}      # log(默认兜底) / aliyun
  aliyun:
    access-key-id: ${ALIYUN_SMS_AK:}
    access-key-secret: ${ALIYUN_SMS_SK:}
    sign-name: ${ALIYUN_SMS_SIGN:}                  # 赠送签名,例如"速通互联验证码"
    template-code: ${ALIYUN_SMS_TEMPLATE:100001}    # 登录/注册模板,赠送,免审核
```

## 阿里云 API 关键参数

**`SendSmsVerifyCode`**(发送,系统生成验证码):

- 必填:`phoneNumber`、`signName`、`templateCode`
- 常用可选:`interval`(发送冷却,秒)、`validTime`(有效期,秒)、`returnVerifyCode`(默认 `false`,不要把明文码返回后端)

**`CheckSmsVerifyCode`**(核验):

- 入参:`phoneNumber`、`verifyCode`、`countryCode`(`cn`)、`outId`(发送时的流水号,可空)
- 返回:`verifyResult`(通过/不通过),据此返回布尔

## 风险与注意

- **SDK 名称**:务必是 `dypnsapi20170525`(号码认证),误用 `dysmsapi20170525`(短信服务)会 endpoint 与 API 全错。
- **签名模板约束**:赠送签名必须搭配赠送模板,自定义签名需报备且可能下发失败,统一用赠送资源。
- **号码范围**:仅支持中国大陆手机号(86),与项目现状一致。
- **AK 安全**:用 RAM 子账号 AK,授予号码认证服务最小权限,生产走环境变量注入,勿入库或入仓。
- **不向后端返回明文验证码**:`returnVerifyCode=false`,核验一律走 `CheckSmsVerifyCode`。
